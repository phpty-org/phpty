# MPL-2.0 uniformly across every PhPty-authored module

Every module PhPty authors ships under the Mozilla Public License 2.0, the same
licence for all of them. This supersedes [ADR-0004](./0004-licensing-varies-by-module.md),
which set a per-module split with Tty, Pty, VTerm and ScreenTest under MIT.

The `LICENSE` file in each module holds the authoritative MPL-2.0 text — Exhibit B,
"Incompatible With Secondary Licenses", included — and travels with the module
when it is split out. The root [`LICENSE.md`](../../LICENSE.md) is the structured
document that describes the scheme and points at those files.

## What forced the reconsideration

The split was never worth its inconsistency. The module `LICENSE` files already
carried the full MPL-2.0 text, while `composer.json` in each declared `MIT` and
the READMEs said MIT — the same module claiming two different licences. v0.1.0
went out in that state. It was a declaration mismatch and nothing more: nothing
was registered on Packagist under MIT, so there is no downstream that relied on
an MIT grant to unwind. The owner's decision is to resolve it towards MPL-2.0,
uniformly, from the next release.

MPL-2.0 also fits the shape of the work better than MIT did: it is file-level
copyleft, so contributions back to a PhPty file stay open without the licence
reaching across FFI into libghostty-vt or into anyone's larger application.

## The reline exception carries over

Reline is the one module this cannot cover, for the reason [ADR-0004](./0004-licensing-varies-by-module.md)
already gave. Upstream reline declares `license = 'Ruby'` — dual
Ruby-licence/2-clause-BSDL, © Yukihiro Matsumoto — and vendors rb-readline under
BSD (© 2009 Park Heesob). A port is a derivative work and cannot be relicensed at
will. If reline is ever ported here it keeps those upstream terms, not MPL-2.0.

libghostty-vt (MIT, © 2024 Mitchell Hashimoto and Ghostty contributors) is
`dlopen`ed at runtime and never redistributed, so it is a dependency, not an
upstream; its MIT licensing is untouched by this decision.

## Consequences

- Any new PhPty-authored module starts under MPL-2.0. There is no per-module
  licence question to answer anymore, except the standing reline exception.
- A module's `composer.json` `license` field must read `MPL-2.0` and match its
  `LICENSE` file. The v0.1.0-era mismatch is the failure this rules out.
- MPL-2.0's file-level notices and the Exhibit B "Incompatible With Secondary
  Licenses" semantics apply to every covered file. New files inherit them.
- The reference implementations under `references/` are submodules and stay under
  their own upstream licences, unaffected.
